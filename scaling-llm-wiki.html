Origins

The LLM Wiki Pattern

In April 2026, Andrej Karpathy posted about using LLMs to build knowledge bases instead of just generating code. The post went viral — 16M+ views, 5,000+ GitHub stars.

Three-Layer Architecture

1
raw/
Immutable sources — articles, docs, notes
2
wiki/
LLM-generated entity, concept, source pages
3
CLAUDE.md
Schema defining page format & workflows

Three Operations

Ingest

Process new source → create/update wiki pages → cross-reference with [[wikilinks]]

Query

Ask questions → synthesize answers from wiki pages → optionally file as analysis

Lint

Health check → find contradictions, orphans, stale info → suggest fixes

Source: gist.github.com/karpathy — llm-wiki
02 / 12
Key Insight

Why It Compounds

Traditional RAG

Rediscovers knowledge from scratch on every question

  • Upload files → chunk → embed → retrieve at query time
  • No accumulation — context window resets each session
  • Requires vector database, extraction pipeline, ontology

LLM Wiki

Persistent, compounding artifact that grows smarter over time

  • Knowledge compiled once, kept current — not re-derived
  • Explicit cross-references via [[wikilinks]] = knowledge graph in plain text
  • No graph database needed — just markdown files in a git repo

"Like a manually-built Graph RAG without the graph database, entity extraction pipeline, or ontology engineering."

Source: levelup.gitconnected.com — Beyond RAG

04 / 12
Scaling Strategy

Three Paths to Scale

01

Git Collaboration

  • Shared repo + PR workflow
  • Branch per person, merge to main
  • Zero cost, full version history

⚠ Merge conflicts on index.md, no real-time collab, no access control

02

Federated Architecture

  • Personal wikis → selective publish → team wiki
  • Team LLM handles dedup & contradiction resolution
  • Confidence scoring per fact, retention curves

⚠ Requires LLM Wiki v2 practices, more complex orchestration

03

RAG + Wiki Hybrid

  • Vector DB for large-scale retrieval
  • LLM Wiki for curated high-value knowledge
  • Governance layer for permissions & audit

⚠ Enterprise-grade, needed when corpus exceeds ~50K-100K tokens

Sources: LLM Wiki v2 · LLM Wiki vs RAG (Atlan)

Path 02 Deep Dive

Federated Architecture

Layer Model
Personal Wiki (private)    Personal Wiki (private)
    ↓ selective publish            ↓ selective publish
    └────────→ Team Wiki (shared) ←────────┘
                             ↓
                  Organization Wiki (org-wide)

Key Principles

  • Each person owns their private wiki
  • Selectively publish pages to team layer
  • Team LLM handles dedup, integration, contradiction resolution

LLM Wiki v2 Additions

Confidence Scoring

Every fact carries a score: how many sources support it, how recently confirmed, whether anything contradicts it

Forgetting / Retention Curves

Facts not accessed or reinforced in months gradually deprioritize — not deleted, but faded

Source Tracking

Every claim links back to sources with dates — contradictions noted explicitly

Reality Check

The Key Challenge

63%

of organizations lack proper data management for reliable knowledge bases

— Gartner

The Real Bottleneck

"Retrieval architecture is the easy part — ungoverned source data is where enterprise knowledge bases fail."

Ingest Cost at Scale

Each source needs 2+ LLM calls per wiki page for routing and synthesis. Manageable for personal use — expensive at enterprise scale (50+ paper corpus ≈ 100-200 API calls).

Source: Atlan — LLM Knowledge Base: Enterprise Use

Accessibility

The UI Layer Problem

The LLM Wiki pattern requires Claude Code for every user. But team members need: open browser → ask question → get answer with citations.

Current Model

Every team member needs:
✗ Claude Code installed
✗ CLI familiarity
✗ API key / subscription
✗ Local repo clone

Does not scale.

Solution: RAG Chat UI Layer

Team members only need:
✓ A web browser
 
The wiki maintainer still uses
Claude Code for ingest/query/lint

Add a read-only RAG UI on top of wiki/

Recommendation

AnythingLLM Architecture

Why AnythingLLM

  • Directly indexes wiki/ folder — Markdown, PDF, 50+ formats
  • Auto chunking → embedding → vector store (built-in LanceDB)
  • Supports Claude API, OpenAI, or Ollama for local/private models
  • Embeddable chat widget for internal portals
  • Docker one-liner deployment

docker pull mintplexlabs/anythingllm
docker run -d -p 3001:3001 \
  -v /path/to/wiki:/app/storage/docs \
  mintplexlabs/anythingllm
