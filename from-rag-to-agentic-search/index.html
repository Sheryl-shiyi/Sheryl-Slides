What is RAG?

Retrieval-Augmented Generation combines an LLM with an external knowledge source. Instead of relying solely on trained parameters, the model retrieves relevant documents before generating an answer.

→

→

Key Insight

RAG separates what the model knows (parametric memory) from what it can look up (non-parametric memory). This was a breakthrough for reducing hallucination.

Lewis et al., "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks," NeurIPS 2020 — arxiv.org/abs/2005.11401

Why RAG Wasn't Enough

Single-Shot Retrieval

One query, one retrieval pass. If the first search misses, the answer is wrong. No retry, no refinement.

Static Pipeline

Data flows in one direction. The system cannot decide to search differently based on what it finds.

Single Source

Typically one vector database. Cannot reason across web, documents, APIs, and databases simultaneously.

No Self-Correction

No mechanism to evaluate retrieval quality. Returns confident answers even from weak evidence.

McKinsey 2026: 58% of enterprise AI teams that deployed standard RAG identified "multi-step reasoning over heterogeneous data sources" as their top limitation.

From Pipeline to Control Loop

BEFORE: RAG Pipeline

Query → Embed → Retrieve → Generate → Answer

Linear. One direction. No feedback. If retrieval fails, the answer fails.

AFTER: Agentic Loop

Interpret → Plan → Retrieve → Evaluate
↓ not good enough? ↓
Rewrite Query → Try Different Source → Re-evaluate
↓ satisfied? ↓
Synthesize → Cite → Verify → Generate

The agent decides when to retrieve, what to retrieve, where to retrieve from, and whether to iterate. This transforms retrieval from a fixed step into an adaptive reasoning process.

Singh et al., "Agentic RAG: A Survey on Agentic RAG," Jan 2025 — arxiv.org/abs/2501.09136

What is Agentic Search?

"RAG retrieves. Agentic search reasons about what to retrieve, evaluates the results, and iterates. The difference is the reasoning loop."

— Boris Cherny, The Agentic Engineering Guide

Agentic RAG

  • Enhances the full RAG pipeline with agents
  • Still centers on retrieval + generation
  • Often uses vector databases
  • Scope: Enterprise knowledge Q&A

Agentic Search

  • Makes the search process itself intelligent
  • No vector DB required
  • Searches heterogeneous sources directly
  • Scope: Deep research, multi-source discovery

In practice, agentic search is often a component within an agentic RAG system.

The Agentic Engineering Guide, Ch. 6 • SoK: Agentic RAG (arXiv:2603.07379, Mar 2026)

Four Agentic Design Patterns

01 Reflection

Agent examines its own output and iterates. Can use a separate "critic agent" for adversarial feedback. The most mature pattern.

02 Tool Use

Agent calls external APIs, code interpreters, databases, web search. The fundamental building block for all other patterns.

03 Planning

Agent decomposes complex tasks into sub-goals. Powerful but less predictable. Enables multi-step reasoning chains.

04 Multi-Agent

Specialized agents collaborate with distinct roles: researcher, reviewer, synthesizer. Extends beyond single-agent loops.

Andrew Ng, "Agentic Design Patterns," DeepLearning.AI / Snowflake BUILD, Mar 2024 • Formalized in arXiv:2601.19752 (Jan 2026)

Anthropic's Approach

"Early versions of Claude Code used RAG + a local vector db, but we found pretty quickly that agentic search generally works better."

— Boris Cherny, creator of Claude Code

Orchestrator-Workers Pattern

Lead Researcher agent spawns parallel subagents, each searching independently. Results written to shared memory, then synthesized and cited.

Three Tracks

Code: agentic search (no vector DB). Docs: Contextual Retrieval (improved RAG, -67% failure). Research: multi-agent orchestrator-workers.

Case Study: RFP Response Skill

A real-world Claude Code skill that uses the Agentic Search pattern to answer batch RFP questions about Red Hat AI products. No vector database, no pre-built index. Agents search raw documentation files and web sources directly.

The Problem

Answering 30-50 RFP questions across 3 products (RHOAI, RHEL AI, RHAII), requiring accurate, cited responses from 5+ documentation repos and public web sources.

Why Not Traditional RAG?

  • Docs update frequently; no time to rebuild vector index
  • Questions span 3 distinct products
  • Answers need cross-product synthesis
  • Local .adoc files + web: heterogeneous sources

Workflow: Agentic Patterns in Action

PLANNING

Scope & Discover — Select target products, auto-discover local doc repos (3-tier: find → ask user → web fallback), group questions by theme.

TOOL USE

Parallel Research — Fan-out agents searching local .adoc files (grep, find, read) and web (redhat.com, docs.redhat.com). Each agent returns raw findings independently.

MULTI-AGENT

Synthesize — Main agent merges findings from all research agents, attributes capabilities to correct products, resolves contradictions.

REFLECTION

Review — Reviewer agent checks technical accuracy, logical consistency, scope leakage, and terminology. Corrections are applied iteratively.

All four of Andrew Ng's agentic design patterns are present in a single workflow. The skill also uses calibration (draft Group 1 first for user feedback) and incremental file saves to handle long sessions.

Key Takeaways

  • 01   RAG evolved from static pipelines to autonomous agent-driven systems in just 5 years (2020-2025).
  • 02   Agentic Search = reasoning loop over search. No vector DB required. Agent decides what, where, and how to search.
  • 03   The Fan-Out/Fan-In pattern (parallel agents + synthesis) is the dominant architecture in 2026.
  • 04   All four agentic patterns (reflection, tool use, planning, multi-agent) compose naturally in real-world systems.
